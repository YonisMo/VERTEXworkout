## 0. ملاحظة على ترقيم المرحلة

المراحل التقنية الأساسية (البنية، البيانات، الواجهة، الأمان، الأداء، SEO، الاختبار، النشر، المراقبة) مكتملة ومعتمدة بالكامل حتى المرحلة 15. المرحلة 16 المنطقية التالية هي **توحيد كيفية توثيق كل هذا** بشكل حي ومستدام قبل الانتقال لمرحلتي **17: Launch & Go-Live Readiness** و**18: Development (البرمجة الفعلية)** — لأن أي مشروع بهذا الحجم من الوثائق (16 وثيقة معتمدة حتى الآن) يحتاج قواعد واضحة لإبقائها **حية ودقيقة** أثناء التطوير الفعلي، لا أن تصبح أرشيفًا تاريخيًا يفقد الثقة به الفريق خلال أسابيع.

## 1. Documentation Architecture — الفلسفة العامة

**المبدأ الأساسي:** **Documentation as Code** — التوثيق يعيش في نفس المستودع (`docs/`)، يُراجَع عبر نفس Pull Requests، ويخضع لنفس معايير الجودة مثل الكود تمامًا. لا توثيق في أداة خارجية منفصلة (Google Docs، Notion) يفقد التزامن مع الكود الفعلي بمرور الوقت.

```
docs/
├── PRD.md                          # معتمد — Phase 1
├── architecture/                    # كل وثائق البنية المعمارية المعتمدة
├── database/                        # Database Schema + الامتدادات الإضافية
├── api/                              # API Design + openapi.json المولَّد آليًا
├── security/                         # Security Architecture + Runbooks الحوادث
├── performance/                       # Performance Strategy
├── seo/                               # SEO Strategy
├── testing/                           # Testing Strategy
├── deployment-guide.md                # Deployment Strategy (نسخة نهائية v2.0)
├── monitoring/                        # Monitoring & Observability Strategy
├── coding-standards.md
├── roadmap.md
├── adr/                               # Architecture Decision Records
├── runbooks/                          # ⭐ جديد — إجراءات تشغيلية خطوة بخطوة للحوادث
├── onboarding/                        # ⭐ جديد — دليل انضمام المطورين الجدد
└── style-guide.md                     # ⭐ جديد — معيار كتابة كل وثيقة جديدة
```

**قاعدة الملكية:** كل وثيقة لها **مالك واضح** (الفريق أو الدور المسؤول عن دقتها)، وليست "ملكًا للجميع وبالتالي لا أحد" — التوثيق الذي لا يملكه أحد تحديدًا يصبح قديمًا بسرعة.

## 2. أنواع التوثيق — تصنيف حسب الجمهور المستهدف

| النوع | الجمهور | مثال | مكان التخزين |
|---|---|---|---|
| **Architectural Documentation** | مطورو الفريق، مراجعو الكود | كل الوثائق المعتمدة من Phase 1 إلى 15 | `docs/architecture/`, `docs/database/`, إلخ |
| **API Documentation** | مطورو تكاملات خارجية (Mobile مستقبلاً، شركاء) | `openapi.json` + Swagger UI | `docs/api/` + `docs.vertexworkout.com` |
| **Operational Documentation (Runbooks)** | فريق On-Call أثناء حادثة فعلية | "ماذا أفعل بالضبط عند SEV-1 في الدفع؟" | `docs/runbooks/` |
| **Developer Onboarding** | مطوّر جديد ينضم للفريق | "كيف أُشغِّل المشروع محليًا من الصفر؟" | `docs/onboarding/` |
| **Inline Code Documentation** | أي مطوّر يقرأ الكود مباشرة | تعليقات JSDoc، `README.md` لكل حزمة/تطبيق | داخل الكود نفسه |
| **End-User Facing Documentation** | العملاء النهائيون (لاحقًا، Phase 2+) | "كيف أستبدل منتجًا؟"، الأسئلة الشائعة | خارج نطاق المستودع التقني، على الموقع نفسه (`/help` مستقبلاً) |

**قاعدة الفصل:** لا يُخلَط توثيق معماري تقني (لماذا اخترنا Supabase) مع توثيق تشغيلي (ماذا أفعل الآن أثناء حادثة) — الأول يُقرأ نادرًا وبتمعن، الثاني يُقرأ تحت ضغط وقت وينبغي أن يكون خطوات مباشرة قابلة للتنفيذ فورًا دون تفكير إضافي.

## 3. Architecture Decision Records (ADRs) — سياسة التوسع

بناءً على الأساس المُعتمَد (5 ADRs في Frontend Architecture)، **السياسة الرسمية لإضافة ADR جديد** من الآن فصاعدًا أثناء التطوير الفعلي:

**متى يُكتَب ADR جديد إلزاميًا:**
- أي اختيار مكتبة/أداة جديدة ليست موثَّقة في المراحل المعتمدة (مثال: اختيار مكتبة رسوم بيانية بديلة لاحقًا).
- أي انحراف عن قرار معماري معتمَد سابقًا (نادر، ويتطلب تبريرًا قويًا لماذا القرار الأصلي لم يعد مناسبًا).
- أي قرار بديل بين خيارين متكافئين ظاهريًا (مثال: اختيار مزوّد بريد إلكتروني محدَّد لـ `packages/email`).

**القالب الموحَّد لكل ADR جديد (نفس نمط الخمسة الأولى، بلا تغيير):**
```
## ADR-00XX: [عنوان القرار]
**الحالة:** معتمد / قيد النقاش / مُستبدَل بـ ADR-00YY
**السياق:** ما المشكلة التي نحلّها؟
**القرار:** ماذا اخترنا؟
**البدائل المدروسة:** جدول: البديل | لماذا لم يُختَر
**النتائج المتوقعة:** ✅ إيجابيات، ⚠️ سلبيات مقبولة
```
**الترقيم:** تسلسلي دائمًا (`0006`, `0007`...)، لا يُعاد استخدام رقم مُستبدَل — ADR قديم يُستبدَل يبقى موجودًا بحالة "مُستبدَل" (Superseded) وليس محذوفًا، لأنه يوثِّق **لماذا** كان القرار القديم منطقيًا وقتها.

## 4. Runbooks — إجراءات الاستجابة التشغيلية

**الفرق عن Incident Response Plan (Security Architecture، القسم 31):** خطة الاستجابة تشرح **المنهجية العامة** (5 مراحل، الأدوار)؛ الـ Runbook يشرح **الخطوات الحرفية التقنية** لسيناريو محدَّد بالضبط — أول ما يُفتَح فعليًا أثناء الحادثة، لا وقت لقراءة فلسفة عامة تحت ضغط.

**مثال Runbook فعلي:**
```markdown
# Runbook: فشل بوابة الدفع Paymob (SEV-1)

## الأعراض
- ارتفاع معدل `PAYMENT_FAILED` في Sentry
- تنبيه Slack من webhook signature failures

## خطوات فورية (بالترتيب، دون تخطي)
1. تحقق من حالة Paymob عبر status.paymob.com
2. إن كانت Paymob نفسها معطلة: فعّل رسالة "الدفع غير متاح مؤقتًا" عبر Feature Flag `payment_paymob_enabled = false`
3. إن كانت المشكلة من جهتنا: راجع آخر نشر في Audit Log (`docs/deployment/deployment-strategy.md`، القسم 12)
4. طبّق Rollback Decision Matrix (Deployment Strategy، القسم 9.2)
5. أبلغ #incidents بالتنسيق المحدَّد (Deployment Strategy، القسم 19.2)

## من يُتصَل به
- Technical Lead (أساسي) → [اسم]
- مسؤول الدفع لدى Paymob (احتياطي) → [رقم/بريد]

## معايير الإغلاق
- معدل PAYMENT_FAILED عاد لمستواه الطبيعي (< 0.5%) لمدة 30 دقيقة متواصلة
```

**قائمة Runbooks الإلزامية عند الإطلاق (المرتبطة بمستويات SEV-1/SEV-2 من Monitoring Strategy):**
| Runbook | يغطي |
|---|---|
| فشل بوابة الدفع | Paymob/Stripe معطلة أو رافضة |
| فشل قاعدة البيانات | Connection Pool مستنزَف، استعلامات بطيئة جماعية |
| اختراق أمني مؤكَّد | تفعيل Incident Response الكامل خطوة بخطوة |
| فشل نشر (Rollback) | تنفيذ Instant Rollback فعليًا خطوة بخطوة على Vercel |
| توقف خدمة البريد الإلكتروني | تأكيدات الطلبات لا تصل، خطوات التحقق والتبديل لمزوّد احتياطي |
