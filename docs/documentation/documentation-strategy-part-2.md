## 5. Developer Onboarding Guide

**الهدف:** مطوّر جديد ينضم للفريق يصل لأول Pull Request مُدمَج **خلال يوم عمل واحد**، دون الحاجة لسؤال زميل عن كل خطوة أساسية.

```
docs/onboarding/
├── 01-getting-started.md      # استنساخ المستودع، pnpm install، supabase start، pnpm dev
├── 02-project-structure.md    # ملخص مبسّط يشير لـ Project Structure الكامل عند الحاجة لتفاصيل أعمق
├── 03-coding-workflow.md       # Branch naming، Commit convention، Pull Request template
├── 04-common-tasks.md         # "كيف أضيف صفحة جديدة؟"، "كيف أضيف حقلًا لقاعدة البيانات؟" — وصفات جاهزة خطوة بخطوة
└── 05-who-to-ask.md            # جدول: الموضوع → الشخص المسؤول (بدل البحث العشوائي في Slack)
```

**معيار "وصفة جاهزة" (Recipe) في `04-common-tasks.md`:** كل مهمة متكررة شائعة تُوثَّق كخطوات دقيقة قابلة للنسخ مباشرة، مثال:
```markdown
## كيف أضيف صفحة جديدة لقسم "المتجر"؟
1. أنشئ `apps/web/src/app/[locale]/(marketing)/store/new-page/page.tsx`
2. أضف الترجمة في `messages/ar/store.json` و`messages/en/store.json`
3. إن احتاجت بيانات: أضف دالة في `packages/api/src/store/`
4. أضف رابطًا داخليًا من صفحة ذات صلة (Internal Linking Strategy، SEO Strategy القسم 9)
5. أضف اختبار E2E إن كانت ضمن المسارات الحرجة (Testing Strategy، القسم 5)
```
**الفائدة:** يقلل الاعتماد على "المعرفة الشفهية" غير المُدوَّنة التي يملكها فقط من بنى النظام أصلاً — أي عضو جديد يستطيع تنفيذ مهمة شائعة بثقة من أول يوم.

## 6. Style Guide الموحَّد لكتابة أي وثيقة جديدة

كل وثيقة مستقبلية (Runbook جديد، ADR جديد، تحديث لوثيقة معتمدة) تلتزم بنفس الأسلوب المُستخدَم عبر كل الوثائق الـ16 المعتمدة حتى الآن:

| العنصر | المعيار |
|---|---|
| **البنية** | عنوان رئيسي → مقدمة موجزة (الغرض والسياق) → أقسام مرقَّمة → جداول للمقارنات → أمثلة كود عند الحاجة → قسم "لماذا" لكل قرار غير بديهي |
| **الجداول قبل الفقرات الطويلة** | أي مقارنة بين خيارين أو أكثر تُعرَض كجدول، وليس فقرة نثرية طويلة يصعب مسحها بصريًا (Scanning) |
| **"Common Mistake يُتجنَّب"** | كل قسم تقني حساس يذكر صراحة الخطأ الشائع الذي يجب تفاديه، وليس فقط "الطريقة الصحيحة" بمعزل عن سياق الخطأ |
| **بدون حشو** | لا فقرات تكرر ما قيل بصياغة مختلفة لمجرد زيادة الطول — كل جملة تحمل معلومة جديدة فعلية |
| **الروابط الداخلية** | أي إشارة لقرار موثَّق في مكان آخر تذكر اسم الوثيقة والقسم بدقة (مثال: "Security Architecture، القسم 12")، وليس إشارة غامضة "كما ذكرنا سابقًا" |
| **اللغة** | عربية فصحى تقنية واضحة، مع الإبقاء على المصطلحات التقنية الإنجليزية كما هي دون ترجمة قسرية (مثل Server Components، RLS) — الترجمة القسرية لمصطلحات تقنية معروفة عالميًا تُربك القارئ التقني أكثر مما تُفيده |

## 7. إدارة إصدارات التوثيق (Documentation Versioning)

| نوع الوثيقة | آلية الإصدار |
|---|---|
| وثائق المراحل المعتمدة (Frozen) | رقم إصدار صريح في الترويسة (مثال: "الإصدار 2.0")؛ أي تعديل حقيقي يرفع الرقم ويُسجَّل السبب في أعلى الوثيقة |
| Runbooks | تُحدَّث فور اكتشاف أن خطوة معينة لم تعد دقيقة (بعد كل حادثة فعلية، مراجعة: "هل الـ Runbook كان دقيقًا؟") |
| ADRs | لا تُعدَّل أبدًا بعد الاعتماد — قرار جديد يُبطل قرارًا قديمًا يُنشئ ADR **جديدًا** يُشير للقديم كـ "مُستبدَل"، وليس تعديلًا للقديم نفسه (يحافظ على السياق التاريخي لماذا اتُّخذ كل قرار وقتها) |
| `openapi.json` | مُولَّد آليًا من الكود (API Design، القسم 27) — لا تعديل يدوي إطلاقًا، يعكس الكود الفعلي دائمًا |

## 8. Diagram-as-Code Standard

كل مخطط في أي وثيقة مستقبلية (ERD جديد، تدفق مستخدم جديد، معمارية جديدة) يُكتَب بصيغة **Mermaid** (نص عادي داخل الملف نفسه)، **وليس صورة خارجية مرفوعة**:

| السبب | التفصيل |
|---|---|
| **قابلية المراجعة (Diffable)** | تغيير في المخطط يظهر كتغيير نصي واضح في Pull Request، تمامًا مثل أي تغيير كود — مراجع الكود يرى **بالضبط** ماذا تغيّر في العلاقة أو الخطوة |
| **لا اعتماد على أداة تصميم خارجية** | أي مطوّر يعدّل المخطط مباشرة في الملف، دون فتح Figma/Draw.io وإعادة تصدير صورة يدويًا |
| **يبقى متزامنًا مع الكود** | مخطط كصورة يصبح قديمًا بصمت (لا أحد يلاحظ أنه لم يعد يعكس الواقع)؛ مخطط كنص يُراجَع كجزء طبيعي من مراجعة أي تغيير معماري |

هذا هو نفس المعيار المُطبَّق فعليًا منذ Database Schema (ERD) وUser Flow وTesting/Deployment (الرسوم التخطيطية النصية) — يُثبَّت الآن كقاعدة رسمية لكل مخطط مستقبلي بلا استثناء.

## 9. End-User Facing Documentation (استعداد لـ Phase 2)

غير مطلوبة في Phase 1 (لا حاجة الآن)، لكن موثَّقة كقرار مستقبلي واضح:
- **مركز مساعدة (`/help`)** يضم أسئلة شائعة حول: الشحن، الإرجاع، الاشتراك في البرامج، حسابي.
- **يُبنى فوق نفس نظام `content_items`** (Database Schema) بنوع محتوى جديد `help_article` — لا حاجة لجدول أو نظام منفصل، امتداد طبيعي للبنية العامة الموجودة فعلاً.
- **لغة مختلفة تمامًا عن التوثيق التقني:** بسيطة، ودّية، خالية من المصطلحات التقنية، موجَّهة لعميل عادي وليس مطوّرًا.
