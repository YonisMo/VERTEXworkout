## 7. Accessibility Testing

| المستوى | الأداة | التكرار |
|---|---|---|
| **آلي أثناء التطوير** | `eslint-plugin-jsx-a11y` (Frontend Architecture، القسم 47) | كل Commit |
| **آلي على مستوى المكوّن** | `@storybook/addon-a11y` (فحص كل Story في Storybook) | كل Pull Request يمس `packages/ui` |
| **آلي على مستوى الصفحة الكاملة** | `axe-playwright` مدمج ضمن اختبارات E2E — يفحص كل صفحة حرجة تلقائيًا عن مخالفات WCAG 2.2 AA أثناء تشغيل نفس سيناريوهات Playwright | كل تشغيل CI |
| **يدوي (لا غنى عنه)** | اختبار فعلي بلوحة المفاتيح فقط (بدون فأرة) لكل صفحة جديدة، واختبار بقارئ شاشة حقيقي (NVDA أو VoiceOver) للمسارات الحرجة | قبل كل إطلاق رئيسي (Major Release) |

```ts
// tests/e2e/accessibility.spec.ts
test('صفحة المتجر خالية من مخالفات WCAG الحرجة', async ({ page }) => {
  await page.goto('/ar/store');
  const results = await new AxeBuilder({ page }).withTags(['wcag2a', 'wcag2aa']).analyze();
  expect(results.violations).toEqual([]);
});
```
**Common Mistake يُتجنَّب:** الاكتفاء بالفحص الآلي فقط (يكتشف ~30-40% من المشاكل الحقيقية كما ورد في Security Architecture) — الفحص اليدوي دوريًا إلزامي وليس اختياريًا.

## 8. API, Security & Performance Testing (تكامل مرجعي)

هذه الأنواع موثّقة بالتفصيل الكامل في مراحلها المخصصة، وتُدمَج هنا **كجزء من نفس CI Pipeline الموحَّد** وليست معزولة:

| النوع | المرجع الكامل | التكامل هنا |
|---|---|---|
| API Testing (Contract Testing مقابل OpenAPI) | API Design، القسم 28 | يُشغَّل ضمن مرحلة `test` في CI بعد Unit/Integration |
| Security Testing (SAST, Secret Scanning, RLS Testing) | Security Architecture، القسم 33 | يُشغَّل كمرحلة CI منفصلة **قبل** أي Deploy، تفشل الـ Pipeline بالكامل عند اكتشاف ثغرة حرجة |
| Load/Stress Testing | Performance Strategy، القسوم 23-24 | **لا** يُشغَّل مع كل PR (بطيء ومكلف) — يُشغَّل يدويًا/مجدولًا أسبوعيًا على بيئة Staging فقط |

**مبدأ التكامل:** هذه الوثيقة (Testing Strategy) هي **المايسترو** الذي ينظّم متى وكيف تُشغَّل كل هذه الأنواع معًا ضمن دورة تطوير واحدة متماسكة، بدل تكرار تفاصيلها التقنية المُوثَّقة مسبقًا.

## 9. Test Data Management

يُبنى فوق `packages/testing` (المُعتمَدة في Frontend Architecture):

| الأداة | الاستخدام |
|---|---|
| **Factories** (`packages/testing/factories`) | توليد بيانات اختبار واقعية وعشوائية (`createProductFactory({ overrides })`) عبر `@faker-js/faker` — يمنع تكرار كتابة كائنات وهمية يدويًا في كل ملف اختبار |
| **Fixtures** (`packages/testing/fixtures`) | بيانات ثابتة مُعرَّفة مسبقًا لسيناريوهات مُتكررة ومحدَّدة (مثال: "مستخدم أدمن جاهز"، "منتج نفدت كميته") |
| **Seed Scripts** (`scripts/seed.ts`) | تعبئة قاعدة بيانات الاختبار/التطوير المحلية ببيانات واقعية كاملة (منتجات، برامج، تمارين) لتجربة التطبيق يدويًا بحالة قريبة من الإنتاج |
| **تنظيف تلقائي بعد كل اختبار (Teardown)** | كل اختبار Integration/E2E يبدأ بقاعدة بيانات نظيفة (Transaction Rollback أو إعادة Seed) — **لا اعتماد اختبار على ترتيب تنفيذ اختبار آخر** (Test Isolation إلزامي) |

## 10. Mocking Strategy

| متى نستخدم Mock | متى **لا** نستخدم Mock |
|---|---|
| خدمات خارجية حقيقية أثناء الاختبار (Paymob API، خدمة البريد، Sentry) — عبر **MSW** (Mock Service Worker، مُجهَّز في `packages/testing/mocks`) | استعلامات قاعدة البيانات الفعلية في اختبارات Integration — تُستخدم قاعدة بيانات اختبار حقيقية (القسم 4) لأن Mock هنا يُخفي أخطاء RLS/Constraints حقيقية |
| الوقت (`Date.now()`) في اختبارات تعتمد على توقيت | منطق الأعمال الداخلي في `packages/api` نفسه — يُختبَر مباشرة، لا يُعزَل بـ Mock وهمي يُفقد الاختبار قيمته الحقيقية |

**قاعدة ذهبية:** **Mock الحدود الخارجية للنظام فقط** (APIs خارجية، الوقت، الشبكة)، **وليس** المنطق الداخلي الذي نملك التحكم الكامل فيه ونريد التأكد فعليًا من صحته.

## 11. CI Test Orchestration

```yaml
# .github/workflows/ci.yml — تسلسل مراحل الاختبار
jobs:
  1_lint_and_types:      # الأسرع، يفشل أولاً لو فيه مشاكل واضحة
  2_unit_tests:          # يعتمد على (1) - Turborepo يشغّل فقط الحزم المتأثرة
  3_security_scan:       # SAST + Secret Scanning - متوازي مع (2)
  4_integration_tests:   # يعتمد على (2) - يحتاج Supabase Local
  5_build:               # يعتمد على (4)
  6_e2e_tests:           # يعتمد على (5) - الأبطأ، يعمل على Preview Deployment فعلي
  7_visual_regression:   # Chromatic - متوازي مع (6)
  8_deploy_preview:      # يعتمد على (6) و(7) بنجاح
```
**قاعدة الترتيب:** المراحل الأسرع والأرخص (Lint, Unit) تُشغَّل أولاً وتفشل بسرعة (**Fail Fast**) — لا داعٍ لانتظار E2E البطيء لو كان هناك خطأ Type بسيط واضح من الثانية الأولى.

## 12. Coverage Targets

| الحزمة/المستوى | الهدف | لماذا هذا الرقم تحديدًا |
|---|---|---|
| `packages/api` | 90%+ | منطق أعمال حرج، يستحق تغطية شبه كاملة |
| `packages/validation` | 100% | مخططات صغيرة محدَّدة، لا عذر لتغطية جزئية |
| `packages/utils` | 100% | دوال نقية بسيطة، سهلة الاختبار الكامل |
| مكونات UI (`packages/ui`) | 70%+ (سلوكي، ليس بصريًا — ذلك عبر Chromatic) | التركيز على السلوك التفاعلي، لا كل تفصيل بصري ثانوي |
| Overall Project | 80%+ (وليس 100%) | **100% تغطية ليست هدفًا صحيًا بذاته** — تدفع أحيانًا لكتابة اختبارات شكلية عديمة القيمة فقط لرفع الرقم؛ 80% مع تركيز على المسارات الحرجة أكثر قيمة فعليًا |

**Common Mistake يُتجنَّب:** ملاحقة نسبة تغطية 100% كهدف بذاته — يُنتج غالبًا اختبارات هشة تتحقق من أسطر كود بسيطة (getters/setters) بدل التركيز على منطق الأعمال الفعلي المعقّد الذي يستحق الاهتمام.

## 13. Flaky Test Management

**Flaky Test** = اختبار ينجح أحيانًا ويفشل أحيانًا أخرى **بدون تغيير في الكود** — عدو خطير للثقة في نظام الاختبار بأكمله.

| السياسة | التفصيل |
|---|---|
| **لا تجاهل صامت** | اختبار Flaky **لا يُعاد تشغيله فقط حتى ينجح (`--retry`) دون تحقيق** — يُفتح Issue فورًا لتتبعه |
| **عزل مؤقت موثَّق** | إن لزم تعطيله مؤقتًا (`test.skip`) لحين الإصلاح، يُرفَق تعليق يوضح: تاريخ التعطيل، رقم الـ Issue، والسبب المُشتبَه به |
| **الأسباب الشائعة المُتجنَّبة معماريًا** | انتظارات ثابتة (`waitForTimeout`) بدل انتظار شرط فعلي (`waitForSelector`)؛ اعتماد اختبار على ترتيب تنفيذ اختبار آخر (Test Isolation، القسم 9)؛ بيانات اختبار مشتركة بين اختبارات متوازية تتعارض |
| **مراجعة دورية** | قائمة "Flaky Tests المعطَّلة مؤقتًا" تُراجَع أسبوعيًا — لا تبقى معطَّلة "للأبد" دون متابعة |

## 14. Test Environments

| البيئة | الغرض | البيانات |
|---|---|---|
| **Local** | تطوير يومي، Supabase CLI محلي | Seed Data وهمية كاملة |
| **CI (Ephemeral)** | تشغيل كل الاختبارات لكل Pull Request | قاعدة بيانات جديدة تمامًا لكل تشغيل، تُحذَف بعد الانتهاء |
| **Staging** | اختبار القبول اليدوي (QA) قبل الإنتاج، Load/Stress Testing الأسبوعي | نسخة مُقرَّبة من بيانات الإنتاج (مُجهَّلة الهوية - Anonymized إن كانت بيانات حقيقية) |
| **Production** | لا اختبارات تلقائية تُنفَّذ هنا **أبدًا** (عدا Health Check البسيط) | بيانات حقيقية فقط |

## 15. Manual QA Process & Bug Triage

**عملية QA اليدوية (تكمل الآلية، لا تُلغيها):**
1. كل ميزة جديدة تحصل على **QA Checklist** مخصص (مبني من User Flow ذي الصلة) قبل اعتبارها "جاهزة" — يُنفَّذه عضو فريق مختلف عن كاتب الكود (مبدأ Second Pair of Eyes).
2. اختبار استكشافي (Exploratory Testing) غير مُخطَّط لسيناريوهات غريبة لم تُغطَّ في الاختبارات الآلية.

**تصنيف الأخطاء (Bug Triage):**
| الأولوية | التعريف | زمن الاستجابة المستهدَف |
|---|---|---|
| **P0 - حرج** | يمنع الشراء/الدخول لكل المستخدمين، أو ثغرة أمنية فعلية | فوري (ساعات) |
| **P1 - عالٍ** | يعطّل ميزة رئيسية لفئة من المستخدمين | يوم عمل واحد |
| **P2 - متوسط** | خطأ واضح لكن له حل بديل (Workaround) | ضمن السبرنت الحالي |
| **P3 - منخفض** | تجميلي/طفيف | يُجدوَل في Backlog عادي |

## ✅ Release Testing Checklist (قبل أي إطلاق رئيسي)

- [ ] كل اختبارات Unit وIntegration خضراء 100% (لا استثناءات "معروفة ومقبولة").
- [ ] كل المسارات الحرجة الثمانية في E2E (القسم 5) خضراء على Chromium وWebKit، Desktop وMobile.
- [ ] Visual Regression (Chromatic) بدون تغييرات غير مُوافَق عليها صراحة.
- [ ] Accessibility Testing الآلي بدون مخالفات WCAG حرجة + جولة يدوية بلوحة المفاتيح على المسارات الحرجة.
- [ ] Security Scan (SAST + Secret Scanning + RLS Tests) بدون ثغرات حرجة/عالية غير معالَجة.
- [ ] Load Testing الأسبوعي على Staging يحقق معايير النجاح المحددة في Performance Strategy.
- [ ] لا اختبارات Flaky معطَّلة بدون متابعة موثَّقة ونشطة.
- [ ] Coverage الكلي ≥ 80%، وحزم الأعمال الحرجة (`packages/api`, `packages/validation`) ضمن أهدافها المحددة.
- [ ] QA يدوي كامل على Staging لكل ميزة جديدة أُضيفت في هذا الإصدار.
- [ ] لا P0/P1 مفتوحة وغير مُعالَجة في قائمة الأخطاء الحالية.
