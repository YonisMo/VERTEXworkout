## 21. Encryption Strategy

| البيانات | نوع التشفير | التفصيل |
|---|---|---|
| البيانات أثناء النقل (In Transit) | TLS 1.3 | مفروض على كل اتصال (HTTPS إلزامي، مُدار تلقائيًا عبر Vercel + Supabase) |
| كلمات المرور | Hashing أحادي الاتجاه (bcrypt، عبر Supabase Auth) | **ليس تشفيرًا قابلًا للعكس** — تفصيل كامل في القسم 22 |
| بيانات الدفع (أرقام البطاقات) | **لا تُخزَّن على خوادم VERTEX إطلاقًا** | تُدار بالكامل عبر Paymob/Stripe (متوافقين مع PCI-DSS) — VERTEX يستقبل فقط حالة الدفع (نجاح/فشل) عبر Webhook، لا يلمس رقم البطاقة أبدًا في أي لحظة |
| الملفات الطبية الاختيارية (`client_profiles`) | تشفير أثناء التخزين (At Rest) عبر تشفير Supabase Storage المدمج (AES-256) + وصول Signed URL محدود المدة فقط | حساسية عالية تستدعي طبقة حماية إضافية عن باقي البيانات |
| بيانات قاعدة البيانات بشكل عام (At Rest) | AES-256 (مُدارة تلقائيًا من طبقة Supabase/AWS الأساسية) | لا حاجة لتنفيذ يدوي إضافي — جزء من ضمانات مزود الخدمة |

**قاعدة أساسية:** VERTEXworkout **لا يبني نظام تشفير خاص به مطلقًا** — يعتمد كليًا على معايير الصناعة المُختبرة (TLS, bcrypt, AES-256 عبر مزودين موثوقين) بدل "اختراع" آلية تشفير داخلية، وهو خطأ أمني شائع وخطير (Rolling Your Own Crypto).

## 22. Password Hashing Strategy

**الخوارزمية:** **bcrypt** (مُدارة تلقائيًا داخل Supabase Auth، Cost Factor = 10 كإعداد افتراضي متوازن بين الأمان وزمن المعالجة).

| لماذا bcrypt وليس MD5/SHA؟ | التفصيل |
|---|---|
| **بطيء عمدًا (Slow by Design)** | يُصعِّب هجمات Brute Force وGPU Cracking بشكل كبير مقارنة بخوارزميات Hashing سريعة مثل SHA-256 المصمَّمة أصلاً لسرعة، لا لأمان كلمات المرور |
| **Salt مدمج تلقائيًا** | كل كلمة مرور تحصل على Salt عشوائي فريد تلقائيًا — يمنع هجمات Rainbow Table تمامًا |
| **قابل للتكيّف (Adaptive)** | يمكن زيادة Cost Factor مستقبلاً مع تطور قدرة الأجهزة، دون تغيير الخوارزمية نفسها |

**ما لا يفعله VERTEXworkout إطلاقًا:** لا تُخزَّن كلمة المرور كنص صريح ولو مؤقتًا حتى في السجلات (Logs) أو رسائل الخطأ — أي رسالة خطأ متعلقة بتسجيل الدخول تبقى عامة ("بيانات الدخول غير صحيحة") دون تحديد إن كان الخطأ في البريد أو كلمة المرور (يمنع كشف معلومة "هذا البريد مسجَّل" لمهاجم يحاول التعداد - Account Enumeration).

## 23. Webhook Security (تعميق)

بالإضافة للتحقق من HMAC Signature (موضّح في API Design)، طبقات إضافية:

| الحماية | التفصيل |
|---|---|
| **قائمة IP بيضاء (حيثما أمكن)** | لو وفَّرت Paymob/Stripe نطاقات IP ثابتة لإرسال Webhooks، تُضاف كطبقة تصفية إضافية قبل حتى الوصول لمنطق التحقق من التوقيع |
| **Idempotency Keys** | كل Webhook يُسجَّل بمعرّف الحدث الفريد في جدول مخصص (`webhook_events`) قبل المعالجة — أي محاولة إعادة إرسال لنفس الحدث تُكتشف وتُرفض دون تنفيذ العملية مرتين (مهم جدًا لتحديثات حالة الطلب المالية) |
| **Timeout قصير + استجابة فورية** | Webhook Handler يُرجع `200 OK` فورًا بعد التحقق الأساسي، وتُنفَّذ المعالجة الفعلية (تحديث الطلب، إرسال بريد) بشكل غير متزامن (Background Job/Queue) — يمنع Timeout من جهة بوابة الدفع الذي قد يسبب إعادة إرسال غير ضرورية |
| **سجل كامل لكل Webhook وارد** | حتى لو فشل التحقق من التوقيع — يُسجَّل المحاولة (بدون تنفيذ) لأغراض المراقبة والتحقيق في محاولات تزوير محتملة |

## 24. Rate Limiting & Abuse Prevention

يُبنى فوق الأساس في API Design (القسم 9)، بقواعد سلوكية إضافية لمنع إساءة الاستخدام تحديدًا (وليس فقط الحماية من الحمل الزائد):

| السلوك المشبوه | الإجراء |
|---|---|
| إنشاء حسابات متعددة من نفس IP خلال وقت قصير | تصعيد تدريجي: تفعيل CAPTCHA إجباري بعد الحساب الثاني من نفس IP خلال ساعة |
| محاولات متكررة لتطبيق أكواد خصم غير صحيحة | حظر مؤقت لتطبيق أكواد الخصم من نفس الجلسة بعد 5 محاولات فاشلة |
| طلبات API بأنماط آلية واضحة (سرعة تفوق القدرة البشرية) | تحليل سلوكي بسيط (Bot Detection heuristics) + تفعيل CAPTCHA أو حظر مؤقت |
| محاولات مسح شامل للمنتجات/الطلبات بأرقام تسلسلية (Enumeration Attack) | UUID غير تسلسلي يجعل هذا الهجوم غير عملي أصلاً (حماية بنيوية، لا تعتمد على Rate Limiting فقط) |

## 25. Brute Force Protection

| نوع الهجوم | الحماية |
|---|---|
| Brute Force على كلمة المرور | Rate Limiting صارم (5 محاولات/15 دقيقة) + CAPTCHA بعد المحاولة الثالثة + قفل حساب مؤقت بعد 5 فشل متتالي |
| Credential Stuffing (استخدام بيانات مسرَّبة من مواقع أخرى) | فحص كلمة المرور الجديدة عند التسجيل مقابل قاعدة بيانات كلمات مرور مُخترَقة معروفة (عبر خدمة مثل Have I Been Pwned API) — رفض كلمات المرور الشائعة/المُخترقة مسبقًا |
| Brute Force على رموز إعادة تعيين كلمة المرور | Token عشوائي طويل (32+ بايت، غير قابل للتخمين عمليًا) + صلاحية قصيرة (15 دقيقة) + استخدام واحد فقط |

## 26. DDoS Protection Strategy

| الطبقة | الحماية |
|---|---|
| **Edge Network** | Vercel Edge Network يوفر حماية DDoS أساسية مدمجة على مستوى البنية التحتية (تصفية حركة مرور ضارة قبل وصولها للتطبيق) |
| **Cloudflare (اختياري، موصى به قبل الإطلاق الفعلي)** | طبقة إضافية أمام Vercel لحركة مرور شديدة الضخامة، مع WAF (Web Application Firewall) لتصفية أنماط هجوم معروفة |
| **Rate Limiting على مستوى التطبيق** | خط الدفاع الأخير لحركة مرور تجاوزت الطبقات السابقة (موضّح في القسم 24) |
| **استراتيجية التوسع (Scaling)** | البنية Serverless (Vercel Functions) تتوسع تلقائيًا مع الحمل — لا نقطة فشل واحدة (Single Point of Failure) على مستوى السيرفر |

## 27. Audit Logging (تعميق تنفيذي)

موضّح معماريًا في Database Schema (القسم 7) — هنا التطبيق الأمني الفعلي:

**كل عملية إدارية حساسة تُسجَّل إلزاميًا** (عبر Middleware مركزي على مستوى `packages/api`، وليس يدويًا في كل دالة — يضمن عدم نسيان أي عملية):
```ts
// packages/api/src/with-audit-log.ts
export function withAuditLog(action: string, entityType: string) {
  return (handler: Function) => async (...args: unknown[]) => {
    const result = await handler(...args);
    await createAuditLog({ actorId: getCurrentUserId(), action, entityType, entityId: result.id, newValues: result });
    return result;
  };
}
// الاستخدام: export const updateProduct = withAuditLog('update', 'product')(async (id, data) => {...});
```
**العمليات المُغطاة إلزاميًا:** تعديل/حذف منتج، تعديل حالة طلب، تعديل صلاحيات مستخدم، الوصول لملف طبي (حتى القراءة تُسجَّل لهذا النوع الحساس تحديدًا)، تسجيل دخول أدمن.

## 28. Monitoring & Security Alerts

| الحدث | آلية التنبيه | من يستقبل التنبيه |
|---|---|---|
| محاولات دخول فاشلة متكررة على حساب واحد | فوري عبر Sentry + بريد للمستخدم نفسه ("محاولة دخول مشبوهة على حسابك") | المستخدم + فريق الأمان |
| ارتفاع مفاجئ في معدل 401/403 | تنبيه Slack فوري (قد يشير لمحاولة هجوم منظّم) | فريق تقني |
| فشل التحقق من توقيع Webhook | تنبيه فوري (محاولة تزوير دفع محتملة) | فريق تقني + مالي |
| وصول لملف طبي من حساب أدمن | تسجيل + مراجعة أسبوعية دورية (ليست تنبيهًا فوريًا لكل حالة، بل تدقيق منتظم) | مسؤول الامتثال/الأمان |
| اكتشاف Secret مسرَّب في Commit | إيقاف فوري للـ CI/CD Pipeline + تنبيه عاجل | فريق تقني (أولوية قصوى) |

## 29. Backup Strategy

| البيانات | التكرار | الاحتفاظ | الموقع |
|---|---|---|---|
| قاعدة البيانات (Supabase Postgres) | نسخ احتياطي تلقائي يومي (مدمج في خطة Supabase) + Point-in-Time Recovery | 30 يومًا (قابلة للتمديد حسب الخطة) | مُدارة من Supabase، مُكرَّرة جغرافيًا |
| ملفات Storage (صور، فيديوهات، مستندات) | نسخ احتياطي أسبوعي إضافي يدوي/آلي عبر `scripts/backup.sh` لموقع تخزين ثانوي مستقل (مثل AWS S3 Glacier) | 90 يومًا | مزود مستقل عن Supabase — لا تعتمد كل النسخ الاحتياطية على مزود واحد |
| متغيرات البيئة/الإعدادات | نسخة موثّقة ومُشفَّرة في مدير أسرار منفصل (مثل 1Password/Vault) بعيدًا عن Vercel نفسه | دائم، محدَّث عند كل تغيير | مدير أسرار مؤسسي منفصل |

**اختبار الاستعادة (Restore Testing):** عملية استعادة تجريبية كاملة من نسخة احتياطية تُجرى **كل 3 أشهر** على بيئة معزولة — نسخة احتياطية لم تُختبَر استعادتها فعليًا لا تُعتبَر نسخة احتياطية موثوقة.

## 30. Disaster Recovery Plan

| السيناريو | خطة الاستجابة | الهدف الزمني (RTO) | أقصى فقدان بيانات مقبول (RPO) |
|---|---|---|---|
| تعطل Vercel بالكامل | التبديل لبيئة نشر احتياطية (يدويًا مُجهَّزة مسبقًا على مزود بديل) | < 4 ساعات | لا فقدان (البيانات في Supabase منفصلة عن Vercel) |
| تعطل/فقدان قاعدة بيانات Supabase | استعادة من آخر نسخة احتياطية + Point-in-Time Recovery للحظة الأقرب للحادثة | < 2 ساعة | < 5 دقائق (بفضل Point-in-Time Recovery) |
| اختراق أمني مؤكَّد (Data Breach) | تفعيل Incident Response Plan فورًا (الجزء 4، القسم 31) — عزل النظام المتأثر أولاً قبل أي إجراء آخر | فوري (العزل) | يعتمد على نطاق الاختراق |
| فقدان مفاتيح API لبوابة دفع | تدوير فوري للمفاتيح المتأثرة عبر لوحة تحكم Paymob/Stripe + تحديث Environment Variables | < 1 ساعة | لا فقدان بيانات (المفاتيح فقط، لا بيانات مستخدمين) |
