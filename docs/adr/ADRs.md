# Architecture Decision Records (ADRs) — VERTEXworkout

> كل ADR يوثّق قرارًا معماريًا مهمًا: السياق، القرار، البدائل المدروسة، والنتائج المتوقعة (إيجابية وسلبية).
> في المستودع الفعلي، كل ADR يكون ملفًا منفصلًا داخل `docs/adr/` مرقّمًا تسلسليًا.

---

## ADR-0001: استخدام Turborepo + pnpm workspaces (Monorepo)

**الحالة:** معتمد

**السياق:**
المشروع يحتاج للتوسع مستقبلًا إلى تطبيقات متعددة (Web, Admin, Coach, Mobile) تشترك في نفس الـ Design System، منطق المصادقة، والأنواع، دون تكرار الكود.

**القرار:**
اعتماد بنية Monorepo باستخدام Turborepo لإدارة الـ Task Pipelines والـ Caching، وpnpm workspaces لإدارة التبعيات المشتركة.

**البدائل المدروسة:**
| البديل | لماذا لم يُختر |
|---|---|
| مشاريع منفصلة (Multi-repo) | تكرار كود كبير، صعوبة مزامنة Design System عبر المشاريع |
| Nx | قوي لكنه أثقل وأكثر تعقيدًا مما يحتاجه المشروع حاليًا؛ Turborepo أخف وأنسب لفريق متوسط الحجم |
| Lerna | أداة أقدم، أقل دعمًا نشطًا مقارنة بـ Turborepo حاليًا |

**النتائج المتوقعة:**
- ✅ مشاركة كود فعلية بين كل التطبيقات
- ✅ بناء واختبار ذكي (فقط الحزم المتأثرة)
- ⚠️ منحنى تعلّم أولي بسيط لأعضاء الفريق الجدد على بنية الـ Workspaces

## ADR-0002: استخدام Supabase كقاعدة بيانات وBackend

**الحالة:** معتمد

**السياق:**
المشروع يحتاج قاعدة بيانات علائقية قوية، نظام مصادقة (Auth)، تخزين ملفات (صور المنتجات، فيديوهات التمارين)، مع القدرة على التوسع دون بناء Backend من الصفر في المرحلة الأولى.

**القرار:**
اعتماد Supabase (PostgreSQL + Auth + Storage + Row Level Security).

**البدائل المدروسة:**
| البديل | لماذا لم يُختر (أو يُختار لاحقًا) |
|---|---|
| Firebase | قاعدة بيانات NoSQL أقل ملاءمة للعلاقات المعقدة (منتجات، طلبات، برامج، أدوار) |
| Backend مخصص (Node.js + Express + Postgres) | وقت تطوير أطول بكثير في المرحلة الأولى، لا داعي له الآن |
| PlanetScale + Backend منفصل | يتطلب بناء طبقة Auth وStorage يدويًا، وقت إضافي غير مبرر |

**النتائج المتوقعة:**
- ✅ سرعة تطوير عالية في MVP
- ✅ Row Level Security يوفر أمانًا على مستوى الصف مباشرة في قاعدة البيانات
- ⚠️ اعتمادية على مزود خارجي (Vendor Lock-in) — مخفّفة عبر طبقة `packages/database` التي تعزل تفاصيل Supabase عن باقي النظام

## ADR-0003: استخدام Next.js 14 (App Router)

**الحالة:** معتمد

**السياق:**
المشروع يحتاج SEO ممتاز (متجر، مدونة، أكاديمية)، أداء عالٍ، ودعم i18n كامل (عربي/إنجليزي).

**القرار:**
اعتماد Next.js 14 مع App Router وTypeScript.

**البدائل المدروسة:**
| البديل | لماذا لم يُختر |
|---|---|
| React (Vite) SPA | لا يوفر SSR/SSG المطلوب لـ SEO قوي |
| Remix | بديل قوي لكن نظام Next.js أكثر نضجًا في دعم i18n والتكامل مع Vercel |
| Nuxt (Vue) | يتطلب تغيير كامل للـ Stack التقني المعتاد للفريق (React أنسب هنا) |

**النتائج المتوقعة:**
- ✅ SEO وأداء ممتازين عبر SSR/SSG
- ✅ تكامل سلس مع next-intl لدعم RTL/LTR
- ⚠️ App Router أحدث نسبيًا ويتطلب الانتباه لأفضل الممارسات المتغيرة باستمرار

## ADR-0004: استخدام Zustand لإدارة حالة الواجهة

**الحالة:** معتمد

**السياق:**
الحاجة لإدارة حالة واجهة بسيطة (سلة التسوق، اللغة المختارة، الثيم) دون تعقيد إضافي.

**القرار:**
اعتماد Zustand لإدارة حالة العميل (Client State)، مع فصلها عن حالة السيرفر التي تُدار عبر TanStack Query.

**البدائل المدروسة:**
| البديل | لماذا لم يُختر |
|---|---|
| Redux Toolkit | Boilerplate أكبر بكثير لحاجة بسيطة نسبيًا |
| React Context API فقط | مشاكل أداء مع التحديثات المتكررة (إعادة render غير ضرورية) |
| Jotai | مشابه لكن Zustand أكثر انتشارًا ودعمًا في نظام Next.js حاليًا |

**النتائج المتوقعة:**
- ✅ كود أقل وأبسط، أداء عالٍ
- ✅ سهولة الاختبار (Stores بسيطة كدوال JavaScript)

## ADR-0005: استخدام TanStack Query لإدارة حالة السيرفر

**الحالة:** معتمد

**السياق:**
البيانات القادمة من Supabase (منتجات، برامج، مقالات) تحتاج Caching، إعادة جلب تلقائي، وإدارة حالات التحميل/الخطأ دون كتابة منطق يدوي متكرر.

**القرار:**
اعتماد TanStack Query كطبقة موحّدة لكل تفاعل مع البيانات القادمة من `packages/api`.

**البدائل المدروسة:**
| البديل | لماذا لم يُختر |
|---|---|
| SWR | مشابه في الفكرة، لكن TanStack Query أغنى بالميزات (Mutations، Infinite Queries، DevTools) المفيدة لمتجر إلكتروني |
| Fetch يدوي + useEffect | إعادة اختراع Caching وError Handling يدويًا — غير مبرر |

**النتائج المتوقعة:**
- ✅ Caching ذكي يقلل الطلبات المكررة
- ✅ تجربة مستخدم أفضل (Optimistic Updates لاحقًا في السلة/الطلبات)
- ✅ فصل واضح بين حالة السيرفر (TanStack Query) وحالة الواجهة (Zustand)
