## 1. Deployment Architecture Overview

**المبدأ الأساسي:** النشر عملية **آلية بالكامل، قابلة للتكرار، وقابلة للتراجع الفوري** — لا نشر يدوي عبر الاتصال المباشر بالسيرفر، ولا خطوة واحدة تعتمد على "تذكّر" فرد معيّن لتنفيذها بالترتيب الصحيح.

```
[Git Push] → [CI: Lint/Test/Security/Build] → [Preview Deployment] → [مراجعة/QA] →
[Merge to main] → [Auto-deploy to Staging] → [Health Checks + Smoke Test] →
[Manual Promotion] → [Production Deployment] → [Deployment Verification] →
[Infrastructure Monitoring المكثّف] → [Audit Log للعملية]
```

كل مربّع في هذا التدفق مُفصَّل بالكامل في الأقسام التالية من هذه الوثيقة، بلا استثناء وبلا اختصار.

## 2. البيئات (Environments)

| البيئة | الغرض | النشر | البيانات |
|---|---|---|---|
| **Development (محلي)** | تطوير يومي | يدوي (`pnpm dev`) | Seed Data وهمية |
| **Preview** | مراجعة كل Pull Request بمعزل | تلقائي عند فتح/تحديث PR | نسخة من Staging DB (قراءة فقط للاختبار البصري) |
| **Staging** | اختبار قبول (QA)، Load Testing الأسبوعي | تلقائي عند الدمج في `main` | بيانات مُقرَّبة من الإنتاج، مُجهَّلة الهوية |
| **Production** | المستخدمون الفعليون | يدوي (Promotion من Staging بعد موافقة) | بيانات حقيقية |

**لماذا Preview Deployments لكل PR؟** يسمح لمراجع الكود ولفريق التصميم برؤية التغيير الفعلي **يعمل** على رابط حقيقي قبل الدمج، بدل الاعتماد فقط على قراءة الكود — يكتشف مشاكل بصرية/وظيفية مبكرًا جدًا في الدورة.

## 3. البنية التحتية للنشر — Vercel + Monorepo

كل تطبيق في `apps/` يُنشَر كـ **مشروع Vercel منفصل** مرتبط بنفس المستودع (دعم Monorepo أصيل في Vercel):

| التطبيق | نطاق الإنتاج | نطاق Staging | ملاحظة |
|---|---|---|---|
| `apps/web` | `vertexworkout.com`, `www.vertexworkout.com` | `staging.vertexworkout.com` | الأولوية القصوى — Phase 1 |
| `apps/admin` | `admin.vertexworkout.com` | `staging-admin.vertexworkout.com` | يُفعَّل فعليًا في Phase 3 |
| `apps/coach` | `coach.vertexworkout.com` | `staging-coach.vertexworkout.com` | يُفعَّل فعليًا في Phase 3 |

**تهيئة Vercel لكل مشروع:**
```json
// vercel.json (لكل apps/*)
{ "buildCommand": "cd ../.. && turbo run build --filter=web", "installCommand": "pnpm install", "outputDirectory": "apps/web/.next" }
```
**الفائدة من `--filter`:** Vercel يبني **فقط** التطبيق المطلوب وتبعياته الفعلية من `packages/`، مستفيدًا من Turborepo Remote Caching — لا إعادة بناء كل الـ Monorepo لكل نشر.

## 4. CI/CD Pipeline الكامل (من Commit إلى Production)

```
1. Push/PR              → GitHub Actions: lint + type-check + unit tests (Fail Fast)
2. Security Scan         → SAST + Secret Scanning + Dependency Audit
3. Integration Tests     → Supabase Local + Route Handlers + RLS Policies
4. Build                 → Turborepo build (متأثر فقط) لكل apps/*
5. E2E + Visual          → Playwright على Preview URL الفعلي + Chromatic
6. Preview Deploy        → رابط فريد لكل PR، يُنشر تلقائيًا من Vercel
7. مراجعة بشرية          → Code Review + QA على رابط Preview
8. Merge → main          → Auto-deploy إلى Staging
9. Health Checks         → فحص فوري لحالة الخدمة على Staging (القسم 5)
10. Smoke Test           → على Staging فورًا بعد النشر (Home, Login, Store, Checkout, API Health)
11. QA يدوي كامل         → على Staging (لميزات جديدة)
12. Manual Promotion     → قرار بشري صريح للترقية لـ Production
13. Production Deploy    → Vercel (Atomic Deployment - القسم 6)
14. Deployment Verification → كل الفحوصات الكاملة (القسم 8)
15. Infrastructure Monitoring → مراقبة مكثّفة (القسم 11)
16. Audit Log            → تسجيل عملية النشر بالكامل (القسم 12)
```

## 5. Health Checks

**الغرض:** نقطة فحص موحّدة وموثوقة تُجيب فورًا على سؤال واحد: **"هل الخدمة تعمل فعليًا الآن؟"** — تُستخدم من CI/CD، من أدوات المراقبة الخارجية (Uptime Monitoring)، ومن عملية Deployment Verification نفسها.

```
GET /api/health
```
```json
// استجابة صحية 200
{
  "status": "ok",
  "timestamp": "2026-07-16T10:00:00Z",
  "checks": {
    "database": { "status": "ok", "latencyMs": 12 },
    "storage": { "status": "ok", "latencyMs": 40 },
    "auth": { "status": "ok", "latencyMs": 8 }
  },
  "version": "1.4.2"
}
```
```json
// استجابة غير صحية 503
{
  "status": "degraded",
  "checks": {
    "database": { "status": "error", "message": "connection timeout" },
    "storage": { "status": "ok", "latencyMs": 38 },
    "auth": { "status": "ok", "latencyMs": 9 }
  }
}
```

| نوع الفحص | التفصيل |
|---|---|
| **Shallow Health Check** | يتحقق فقط أن التطبيق نفسه يستجيب (Process حي) — سريع جدًا، يُستخدم من Vercel نفسه لأغراض التوجيه الداخلي |
| **Deep Health Check** (`/api/health`) | يتحقق فعليًا من الاتصال بقاعدة البيانات، Storage، وخدمة المصادقة — كل واحد بفحص فعلي مصغَّر (استعلام بسيط، وليس فقط "افتراض أنه يعمل") |
| **حد زمني صارم (Timeout)** | كل فحص فرعي له مهلة أقصى 2 ثانية — فحص Health لا يجب أن يُعلِّق نفسه بانتظار خدمة متعطلة |
| **تكرار الفحص الخارجي** | أداة Uptime Monitoring خارجية (مثل Better Uptime) تستدعي `/api/health` كل 60 ثانية من عدة مواقع جغرافية، وتُنبِّه فورًا عند 3 فشل متتالي |

**قاعدة صارمة:** `/api/health` **لا يتطلب مصادقة** (متاح علنًا) لكنه **لا يكشف أي معلومة حساسة** — فقط حالة "يعمل/لا يعمل" ورمز الإصدار الحالي، بدون تفاصيل تقنية عن البنية الداخلية.

## 6. Zero-Downtime Deployment

**آلية Vercel (Atomic Deployments):** كل نشر يُبنى بالكامل في بيئة معزولة، ثم يتم التبديل الفوري (Instant Cutover) للنسخة الجديدة فقط بعد نجاح البناء بالكامل — **لا توجد لحظة يكون فيها الموقع "نصف مُحدَّث"**؛ المستخدمون إما يرون النسخة القديمة بالكامل أو الجديدة بالكامل.

**التعامل مع Requests أثناء التبديل:** الطلبات الجارية وقت التبديل تُكمَل على النسخة التي بدأت عليها (Connection Draining) — لا يُقطَع طلب مستخدم منتصف عملية شراء بسبب توقيت نشر متزامن.

**حالة خاصة — تغييرات Database Schema:** يجب أن يكون كل Migration **متوافقًا خلفيًا (Backward Compatible)** مع الكود القديم للحظة الانتقالية القصيرة (موضّح بالتفصيل في القسم 7).

## 7. Database Migration Deployment Strategy

**القاعدة الذهبية:** **الكود الجديد يجب أن يعمل مع قاعدة البيانات القديمة، وقاعدة البيانات الجديدة يجب أن تعمل مع الكود القديم** لفترة انتقالية قصيرة — لأن الـ Migration وBuild الجديد لا يصلان بالضبط في نفس اللحظة عبر كل الـ Instances.

**نمط النشر الآمن (Expand-Contract Pattern):**
```
1. Expand: إضافة عمود/جدول جديد (لا حذف، لا إعادة تسمية مباشرة) → يُنشَر أولاً، متوافق مع الكود القديم والجديد معًا
2. Migrate: الكود الجديد يبدأ باستخدام الحقل الجديد، مع كتابة مزدوجة مؤقتة إن لزم
3. Contract: بعد التأكد من استقرار الكود الجديد لفترة (أيام)، يُحذَف العمود/الحقل القديم في Migration منفصل لاحق
```
**مثال فعلي:** إعادة تسمية `products.title` إلى `products.name` **لا تتم مباشرة** (`RENAME COLUMN`) في نشر واحد — بل: (1) إضافة عمود `name` جديد، (2) نسخ البيانات، (3) الكود الجديد يقرأ من `name`، (4) بعد استقرار كامل، حذف `title` القديم في نشر لاحق منفصل.

**تنفيذ Migrations:** عبر `supabase migration up` ضمن خطوة CI/CD **منفصلة قبل** نشر الكود الجديد فعليًا — الترتيب: DB Migration أولاً (Expand فقط، آمن دائمًا) ثم نشر الكود.

## 8. Deployment Verification

**الفرق عن Health Checks (القسم 5):** Health Checks تُجيب "هل الخدمة حية؟"، بينما Deployment Verification تُجيب سؤالًا أوسع وأكثر تحديدًا: **"هل هذا النشر تحديدًا نجح بكل تفاصيله الوظيفية؟"** — عملية أشمل تُنفَّذ **فقط بعد كل نشر**، وليست فحصًا دوريًا مستمرًا.

**قائمة التحقق الكاملة بعد كل نشر إنتاجي:**

| العنصر | طريقة التحقق | معيار النجاح |
|---|---|---|
| Health Endpoint | استدعاء `/api/health` | `status: "ok"` لكل الفحوصات الفرعية |
| Database | استعلام قراءة وكتابة تجريبي (سجل اختبار مخصص يُحذَف فورًا) | نجاح القراءة والكتابة معًا |
| Storage | رفع واسترجاع ملف تجريبي صغير | نجاح الرفع والاسترجاع خلال أقل من 3 ثوانٍ |
| Payments (Sandbox Ping) | استدعاء نقطة فحص من Paymob/Stripe (وليس معاملة فعلية) | استجابة صحية من بوابة الدفع |
| Emails | إرسال بريد اختباري داخلي عبر `packages/email` | استلام فعلي خلال دقيقتين |
| Search | استعلام بحث تجريبي ثابت (`"vertex"`) | إرجاع نتائج متوقَّعة معروفة مسبقًا |
| Analytics | التحقق أن سكربتات GA4/GTM/Meta Pixel تُحمَّل فعليًا في الصفحة المنشورة | ظهور Network Request صحيح لكل مزوّد |
| Monitoring | التحقق أن Sentry يستقبل حدثًا تجريبيًا مقصودًا | ظهور الحدث في لوحة Sentry خلال دقيقة |
| Smoke Test (Testing Strategy، القسم 16.19) | تشغيل تلقائي كامل | كل الصفحات الست (Home, Login, Dashboard, Store, Checkout, API Health) تستجيب 200 |

**القاعدة الحاسمة:** **لا يُعتبَر النشر ناجحًا ومكتملًا حتى تجتاز كل بنود هذا الجدول بنجاح فعلي موثَّق** — نجاح عملية الـ Build وحده لا يعني أن النشر نجح.

## 9. Rollback Strategy & Rollback Decision Matrix

### 9.1 آليات التراجع حسب نوع المشكلة

| نوع المشكلة | آلية التراجع | الزمن المستهدَف |
|---|---|---|
| **مشكلة في الكود** (بعد نشر Vercel) | **Instant Rollback** — Vercel يحتفظ بكل نشرة سابقة جاهزة؛ التراجع = تبديل فوري للنشرة السابقة الناجحة، بدون إعادة بناء | أقل من دقيقتين |
| **مشكلة في Migration** | تشغيل `Down Migration` المُختبَر مسبقًا (إلزامي لكل Migration، Testing Strategy القسم 16.4) | أقل من 15 دقيقة |
| **مشكلة اكتُشفت بعد Cache انتشر (ISR)** | `revalidatePath`/`revalidateTag` فوري لإبطال المحتوى الخاطئ من الكاش عبر كل Edge Nodes | فوري |
| **اختراق أمني مؤكَّد** | يتبع Incident Response (القسم 18) — العزل أولًا، ثم التراجع | فوري (العزل) |

### 9.2 Rollback Decision Matrix — من يقرر، ومتى، وكيف؟

هذا الجدول يحدد بدقة **متى يجب التراجع فورًا بدون نقاش، ومتى يُقيَّم القرار أولًا**:

| الأثر المُلاحَظ | مثال فعلي | القرار | من يملك الصلاحية |
|---|---|---|---|
| **يمنع الشراء/الدخول لكل المستخدمين** | صفحة الدفع تُعيد 500 لكل محاولة | **Rollback فوري إلزامي، بدون انتظار موافقة** | أي عضو Technical Lead |
| **ثغرة أمنية فعلية مكتشفة** | تسريب بيانات مستخدم آخر عبر IDOR | **Rollback فوري إلزامي + تفعيل Incident Response** | أي عضو Technical Lead |
| **يعطّل ميزة رئيسية لفئة من المستخدمين** | فشل تسجيل الدخول عبر Google تحديدًا فقط | **تقييم سريع (15 دقيقة) ثم قرار** — إصلاح سريع Forward-fix أم Rollback | Technical Lead + Incident Commander |
| **خطأ واضح له حل بديل (Workaround)** | خطأ عرض بصري في صفحة ثانوية | **لا Rollback** — يُصلَح في نشرة تالية عادية ضمن السبرنت | فريق التطوير العادي |
| **تدهور طفيف في الأداء غير حرج** | ارتفاع TTFB بنسبة 10% دون تجاوز الهدف | **مراقبة إضافية فقط، لا إجراء فوري** | فريق الأداء يُتابع دون تدخل طارئ |

**قاعدة القرار الحاكمة:** أي عضو في فريق Technical Lead يملك صلاحية اتخاذ قرار Rollback فوري **بدون** انتظار موافقة إدارية في حالات الصف الأول والثاني من الجدول أعلاه — السرعة أهم من البيروقراطية عند وجود مستخدمين متأثرين فعليًا أو خطر أمني حقيقي.

## 10. Backup & Disaster Recovery

### 10.1 استراتيجية النسخ الاحتياطي التطبيقية على النشر

| البيانات | التكرار | الاحتفاظ | متى تُستخدَم في سياق النشر |
|---|---|---|---|
| قاعدة البيانات (Supabase Postgres) | نسخ احتياطي تلقائي يومي + Point-in-Time Recovery | 30 يومًا | قبل أي Migration كبير الأثر، تُؤخَذ نسخة يدوية إضافية فورية كإجراء احترازي مباشر قبل التنفيذ |
| ملفات Storage | نسخ احتياطي أسبوعي إضافي لموقع تخزين ثانوي مستقل | 90 يومًا | لا علاقة مباشرة بدورة النشر العادية، لكنها خط دفاع أخير عند فشل كارثي مصاحب لنشر معطوب |
| متغيرات البيئة/الإعدادات | نسخة موثّقة ومُشفَّرة في مدير أسرار منفصل | دائم، محدَّث عند كل تغيير | تُراجَع كجزء من كل تغيير في القسم 15 (Environment Secrets) |

### 10.2 إجراء ما قبل أي نشر كبير الأثر (Pre-Deployment Backup)

قبل أي نشر يتضمن Migration من النوع "Contract" (حذف نهائي لعمود/جدول)، يُنفَّذ إلزاميًا:
```
1. نسخة احتياطية يدوية فورية لقاعدة البيانات (بالإضافة للنسخة اليومية التلقائية)
2. التحقق من نجاح النسخة (استعلام تأكيدي على حجم النسخة وسلامتها)
3. توثيق رقم/توقيت هذه النسخة في سجل النشر (Audit Log، القسم 12)
4. عندها فقط يُسمَح بتنفيذ Migration من نوع Contract
```

### 10.3 خطة التعافي من الكوارث المرتبطة بالنشر

| السيناريو | خطة الاستجابة | الهدف الزمني (RTO) | أقصى فقدان بيانات مقبول (RPO) |
|---|---|---|---|
| نشر معطوب أدى لتلف بيانات جزئي | استعادة فورية من النسخة الاحتياطية المُوثَّقة في القسم 10.2 | أقل من ساعتين | أقل من 5 دقائق (بفضل Point-in-Time Recovery) |
| تعطل Vercel بالكامل أثناء النشر | التبديل لبيئة نشر احتياطية مُجهَّزة مسبقًا على مزود بديل | أقل من 4 ساعات | لا فقدان (البيانات في Supabase منفصلة عن Vercel) |
| تعطل/فقدان قاعدة بيانات Supabase أثناء أو بعد النشر مباشرة | استعادة من آخر نسخة احتياطية + Point-in-Time Recovery للحظة الأقرب للحادثة | أقل من ساعتين | أقل من 5 دقائق |

### 10.4 اختبار الاستعادة الدوري

عملية استعادة تجريبية كاملة تُجرى **كل 3 أشهر** على بيئة معزولة، ومحاكاة كاملة لكارثة (Disaster Recovery Drill) تُجرى **كل 6 أشهر** تشمل Database Failure وStorage Failure وAPI Failure وRegion Failure، مع قياس RTO وRPO الفعليين مقارنة بالأهداف المُعلَنة أعلاه — نسخة احتياطية لم تُختبَر استعادتها فعليًا لا تُعتبَر نسخة احتياطية موثوقة.

## 11. Infrastructure Monitoring (بما فيها Post-Deploy Monitoring)

### 11.1 المراقبة المستمرة للبنية التحتية

| الأداة | الغرض | التكرار |
|---|---|---|
| **Vercel Analytics** | Core Web Vitals لمستخدمين حقيقيين، توزيع حسب الصفحة والجهاز | مستمر، لحظي |
| **Sentry Performance Monitoring** | زمن استجابة كل Route Handler/Server Action، تتبع Traces عبر الطبقات | مستمر، لحظي |
| **Supabase Dashboard** | استخدام الـ Connections، الاستعلامات البطيئة، حجم قاعدة البيانات | مستمر، لحظي |
| **Better Uptime (أو مكافئ)** | استدعاء `/api/health` من مواقع جغرافية متعددة | كل 60 ثانية |
| **Vercel Deployment Logs** | سجل كامل لكل عملية بناء ونشر (نجاح/فشل، مدة البناء) | لكل نشر |

### 11.2 المراقبة المكثّفة بعد كل نشر إنتاجي (Post-Deploy Monitoring)

بعد كل نشر إنتاجي، **ساعة مراقبة مكثّفة إلزامية** (وليس ترك النظام "يعمل ونشوف"):

| الدقائق | الإجراء |
|---|---|
| 0-5 | Deployment Verification الكامل (القسم 8) |
| 5-15 | مراقبة مباشرة للوحة Sentry (معدل الأخطاء) وVercel Analytics (Core Web Vitals اللحظية) |
| 15-60 | مراقبة أخف وتيرة، مع تنبيهات تلقائية فقط عند تجاوز عتبات معرَّفة مسبقًا (معدل 500 أكبر من 1%، TTFB أكبر من ثانية واحدة) |

**معيار "النشر مستقر":** لا ارتفاع في معدل الأخطاء 5xx، لا تدهور ملحوظ في Core Web Vitals، Deployment Verification أخضر بالكامل — عندها فقط يُعتبَر النشر ناجحًا نهائيًا ويُغلَق تتبعه.

### 11.3 التنبيهات (Alerting Thresholds)

| المقياس | العتبة | الإجراء |
|---|---|---|
| معدل أخطاء 5xx | أكبر من 1% خلال 5 دقائق | تنبيه Slack فوري لفريق Technical Lead |
| TTFB متوسط | أكبر من ثانية واحدة لمدة 10 دقائق متتالية | تنبيه Slack، مراجعة فورية |
| فشل Health Check | 3 فشل متتالٍ | تنبيه عاجل + بدء تقييم Rollback Decision Matrix |
| استخدام قاعدة البيانات (Connections) | أكبر من 80% من الحد الأقصى | تنبيه استباقي قبل الوصول لحد الاستنزاف الكامل |

## 12. Audit Logs (خاصة بعمليات النشر)

كل عملية نشر إلى بيئة Staging أو Production تُسجَّل إلزاميًا في سجل تدقيق مخصص للنشر، منفصل عن `audit_logs` الخاصة بمنطق الأعمال (Database Schema) لكن بنفس مبدأ عدم القابلية للتعديل (Immutability):

```json
{
  "deploymentId": "dep_9f2a...",
  "environment": "production",
  "app": "web",
  "version": "1.4.2",
  "commitSha": "a3f9c21",
  "triggeredBy": "ahmed@vertexworkout.com",
  "approvedBy": "sara@vertexworkout.com",
  "startedAt": "2026-07-16T09:58:00Z",
  "completedAt": "2026-07-16T10:03:12Z",
  "healthCheckResult": "passed",
  "deploymentVerificationResult": "passed",
  "rollbackOccurred": false
}
```

**الحقول الإلزامية في كل سجل:** من نفَّذ الأمر فعليًا (`triggeredBy`)، من وافق على الترقية للإنتاج (`approvedBy` — يختلف عن المُنفِّذ في أغلب الحالات، تطبيقًا لمبدأ Second Pair of Eyes)، رقم الإصدار وHash الـ Commit الدقيق، ونتيجة كل من Health Check وDeployment Verification.

**الفائدة:** عند أي مشكلة لاحقة، يمكن تحديد بدقة **متى** حدث النشر المسبِّب، **من** نفَّذه ووافق عليه، و**ما الإصدار** بالضبط الذي كان يعمل قبل المشكلة — أساسي لتحقيقات Incident Response (القسم 18) ولـ Rollback الدقيق للإصدار الصحيح تحديدًا.

**الاحتفاظ:** سجلات نشر Production تُحفَظ بشكل دائم (لا تُحذَف أبدًا)، بينما سجلات Preview/Staging تُحفَظ 90 يومًا.

## 13. Release Versioning & Changelog

**الإصدار:** Semantic Versioning (`MAJOR.MINOR.PATCH`) لكل تطبيق (`apps/web@1.4.2`) — يُدار عبر **Changesets** (يتكامل جيدًا مع بنية Monorepo/Turborepo، يسمح بترقية إصدارات مستقلة لكل تطبيق/حزمة).

```bash
pnpm changeset          # عند كل PR يحتوي تغييرًا يستحق توثيقًا في الـ Changelog
pnpm changeset version  # يُحدِّث أرقام الإصدارات + يُولِّد CHANGELOG.md تلقائيًا
```
**قاعدة الإصدار:**
- `PATCH`: إصلاح خطأ لا يغيّر السلوك المُعلَن.
- `MINOR`: ميزة جديدة متوافقة خلفيًا (مثال: صفحة Wishlist جديدة).
- `MAJOR`: تغيير جذري (تغيير في API العام، أو في تجربة أساسية معروفة).

**CHANGELOG.md** يُولَّد آليًا ويُرفَق بكل Release على GitHub — مرجع دائم لتاريخ كل تغيير فعلي في المشروع، وليس تخمينًا لاحقًا.

## 14. Feature Flags Strategy

للميزات كبيرة الأثر (مثال: تفعيل الدفع الفعلي أول مرة، إطلاق قسم جديد من Academy) — نشر الكود **لا يعني** تفعيله فورًا لكل المستخدمين:

```ts
// packages/lib/src/feature-flags.ts
export async function isFeatureEnabled(flag: string, userId?: string): Promise<boolean> {
  // يبدأ بسيطًا (Environment Variable)، يتطور لاحقًا لخدمة مخصصة (LaunchDarkly/Vercel Flags) عند الحاجة
  return getFlagValue(flag, userId);
}
```
**سيناريوهات الاستخدام:**
- **Percentage Rollout:** تفعيل ميزة لـ 10% من المستخدمين أولاً، مراقبة، ثم التوسع تدريجيًا لـ 100%.
- **Kill Switch فوري:** تعطيل ميزة مشكوك بها فورًا (تغيير Flag، بدون نشر كود جديد أو Rollback كامل) — أسرع بكثير من التراجع الكامل عن نشرة بأكملها.

## 17. Cost Monitoring

النشر المتكرر والتوسع التلقائي (Serverless) يعنيان تكلفة متغيّرة يجب مراقبتها بنفس جدية مراقبة الأداء والأمان:

| المصدر | ما يُراقَب | آلية المراقبة |
|---|---|---|
| **Vercel** | استهلاك Function Execution Time، Bandwidth، Build Minutes | لوحة تحكم Vercel Usage + تنبيه عند تجاوز 80% من حد الخطة الحالية |
| **Supabase** | حجم قاعدة البيانات، Storage المُستخدَم، عدد المستخدمين النشطين شهريًا (MAU) | لوحة تحكم Supabase + مراجعة شهرية دورية |
| **خدمات خارجية** (Sentry, Upstash Redis للـ Rate Limiting, خدمة البريد) | الاستهلاك مقابل حدود الخطة المشترك بها | مراجعة شهرية موحّدة لكل الفواتير مجتمعة |
| **تنبيه استباقي** | أي مصدر يقترب من 80% من حد خطته الحالية | تنبيه فوري للفريق التقني/الإداري لاتخاذ قرار الترقية قبل الوصول لحد يوقف الخدمة فجأة |

**قاعدة تصميم مرتبطة بالنشر مباشرة:** أي ميزة جديدة تُقدَّر تكلفتها التشغيلية المتوقعة (استعلامات إضافية، تخزين إضافي، طلبات API خارجية) **قبل** الموافقة على نشرها للإنتاج — لا مفاجآت في الفاتورة بعد الإطلاق.

**مراجعة شهرية موحّدة:** تقرير تكلفة شهري يقارن الإنفاق الفعلي بالمتوقع، ويُحدِّد أي انحراف غير مبرَّر يستحق تحقيقًا (مثال: ارتفاع مفاجئ في استعلامات قاعدة البيانات قد يشير لاستعلام غير مُحسَّن نُشِر مؤخرًا).

## 18. Production Freeze Policy

**التعريف:** فترة زمنية محدَّدة **يُمنَع خلالها أي نشر جديد للإنتاج** إلا لإصلاحات حرجة (Hotfixes) موافَق عليها صراحة من Incident Commander أو Technical Lead.

### 18.1 متى يُفعَّل Production Freeze إلزاميًا

| الحالة | مدة التجميد |
|---|---|
| أثناء إجراء نشر كبير الأثر (Major Migration) وحتى تأكيد الاستقرار الكامل | من بداية النشر حتى انتهاء ساعة المراقبة المكثّفة (القسم 11.2) |
| فترة تسويقية/ترويجية عالية الحركة (حملة إطلاق، عروض كبرى) | من 24 ساعة قبل الحملة حتى 24 ساعة بعد انتهائها |
| أثناء التحقيق في حادثة أمنية أو أداء غير محلولة (Incident Response نشط) | حتى إغلاق الحادثة رسميًا (القسم 18) |
| قبل عطلات رسمية ممتدة (فريق الاستجابة غير متوفر بالكامل) | فترة العطلة + يوم عمل واحد قبلها وبعدها |

### 18.2 ما يُسمَح به أثناء التجميد

- إصلاحات حرجة (Hotfixes) لمشكلة P0/P1 قائمة فعليًا — تمر بنفس CI/CD الكامل، لكن بأولوية قصوى ومراجعة مُعجَّلة (لا تخطي أي خطوة اختبار أو أمان).
- نشر على بيئتي Preview وStaging يبقى مسموحًا بلا قيود — التجميد خاص بـ **Production فقط**.

### 18.3 آلية التفعيل والإلغاء

يُعلَن Production Freeze رسميًا عبر قناة تواصل الفريق (Slack) مع تحديد السبب والمدة المتوقعة، ويُلغى بإعلان صريح مماثل بعد التأكد من زوال السبب — لا تجميد "صامت" غير مُوثَّق يُربك الفريق حول ما هو مسموح فعليًا.

## 19. Incident Response (في سياق النشر)

هذا القسم يُطبِّق خطة Incident Response الشاملة (المُعتمَدة بالكامل في Security Architecture، القسم 31) **تحديدًا على سياق حوادث النشر نفسها** — أي عندما يكون سبب الحادثة نشرًا حديثًا وليس بالضرورة اختراقًا أمنيًا خارجيًا.

### 19.1 مراحل الاستجابة لحادثة ناتجة عن نشر

| المرحلة | الإجراء الخاص بسياق النشر |
|---|---|
| **1. الاكتشاف** | تنبيه من Infrastructure Monitoring (القسم 11) أو فشل في Deployment Verification (القسم 8) |
| **2. الاحتواء** | مراجعة فورية لـ Rollback Decision Matrix (القسم 9.2) لتحديد ما إذا كان التراجع الفوري مطلوبًا |
| **3. الاستئصال** | إن قُرِّر عدم التراجع الفوري (مشكلة متوسطة الأثر)، يُحدَّد الإصلاح المطلوب (Forward-fix) وتُفعَّل Production Freeze (القسم 18) حتى نشره |
| **4. الاستعادة** | نشر الإصلاح عبر نفس CI/CD الكامل، أو تأكيد نجاح الـ Rollback عبر Deployment Verification كاملة من جديد |
| **5. التعلّم** | تقرير Post-Mortem خلال 72 ساعة يُضاف كسجل مرتبط بـ Audit Log (القسم 12) الخاص بهذا النشر تحديدًا |

### 19.2 قناة التصعيد المباشرة

كل حادثة مرتبطة بنشر تُبلَّغ فورًا عبر قناة Slack مخصصة (`#incidents`) بالتنسيق التالي إلزاميًا: رقم/توقيت النشر المتأثر (من Audit Log)، الأثر المُلاحَظ، القرار المتخذ (Rollback أو Forward-fix)، والوقت المتوقع للحل — لا حادثة تُدار عبر رسائل متفرقة غير منظَّمة.

### 19.3 الربط المباشر مع باقي الوثيقة

قرار Incident Response هنا يستهلك مباشرة: Rollback Decision Matrix (القسم 9.2) لتحديد سرعة التصرف، Backup & Disaster Recovery (القسم 10) إن استدعى الأمر استعادة بيانات، وAudit Logs (القسم 12) لتوثيق كل خطوة تُتَّخذ أثناء الحادثة نفسها بدقة زمنية كاملة.

## ✅ Deployment Production Checklist (النسخة النهائية الكاملة)

- [ ] كل بيئة (Preview/Staging/Production) معزولة تمامًا في الأسرار والبيانات — لا تسريب بينها.
- [ ] Zero-Downtime Deployment مُختبَر فعليًا (نشرة تجريبية بدون انقطاع ملحوظ).
- [ ] كل Migration يتبع نمط Expand-Contract، ومُختبَر بـ Up/Down/Rollback (Testing Strategy، القسم 16.4).
- [ ] Health Checks (`/api/health`) تعمل بفحص عميق فعلي لقاعدة البيانات وStorage وAuth، وليس فحصًا سطحيًا فقط.
- [ ] Deployment Verification الكاملة بكل بنودها التسعة (القسم 8) تُنفَّذ تلقائيًا بعد كل نشر إنتاجي.
- [ ] Rollback الفوري (أقل من دقيقتين) مُختبَر فعليًا على Staging مرة واحدة على الأقل قبل الإطلاق الأول.
- [ ] Rollback Decision Matrix (القسم 9.2) موزَّعة ومفهومة من كل فريق Technical Lead، لا غموض حول من يقرر ومتى.
- [ ] نسخة احتياطية يدوية فورية تُؤخَذ قبل أي Migration من نوع Contract، وتُوثَّق في Audit Log.
- [ ] اختبار استعادة فعلي كل 3 أشهر، ومحاكاة كوارث كاملة كل 6 أشهر، مع قياس RTO/RPO الفعليين.
- [ ] Infrastructure Monitoring مفعَّل بكل أدواته الخمسة (Vercel Analytics, Sentry, Supabase Dashboard, Uptime Monitoring, Deployment Logs) مع تنبيهات فعلية مربوطة.
- [ ] ساعة المراقبة المكثفة بعد كل نشر إنتاجي مطبَّقة كإجراء إلزامي موثَّق للفريق، وليس اختياريًا.
- [ ] Audit Log لكل عملية نشر Production يُسجَّل تلقائيًا بكل الحقول الإلزامية (منفِّذ، موافِق، إصدار، نتيجة الفحوصات).
- [ ] Changesets/Versioning مُفعَّل، CHANGELOG.md يتولَّد تلقائيًا وصحيح.
- [ ] DNS وSSL لكل النطاقات الثلاثة (web/admin/coach) مُهيَّأة وتُجدَّد تلقائيًا.
- [ ] فصل الأسرار الصارم بين البيئات مؤكَّد (لا مفتاح Production في أي بيئة أخرى).
- [ ] خطة Feature Flags جاهزة للاستخدام عند أول ميزة كبيرة الأثر (تفعيل الدفع الفعلي).
- [ ] Cost Monitoring مفعَّل مع تنبيهات استباقية عند 80% من حدود كل خطة مستخدَمة.
- [ ] Production Freeze Policy موثَّقة ومفهومة، مع آلية إعلان/إلغاء واضحة عبر Slack.
- [ ] Incident Response في سياق النشر مربوط فعليًا بـ Rollback Decision Matrix وAudit Logs وBackup/DR، وليس عملية منفصلة غير مترابطة.
